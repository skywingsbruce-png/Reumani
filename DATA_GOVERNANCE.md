# 数据治理（硬规则）— Reumani

本文件规定三类数据必须**严格分离**，任何在此仓库工作的人（含 AI 助手）都必须遵守。
违反"评测隔离"会让所有效果数字失去意义。

## 三类数据，绝不混用

| 类别 | 用途 | 存放 | 是否改模型权重 |
|---|---|---|---|
| **1. 模型训练数据** | 微调基础模型权重 | `train/`（当前为空） | 是 |
| **2. 知识库 / 文献数据湖** | RAG 检索、证据综合 | `data_lake/` | 否 |
| **3. 评测集** | 只衡量系统效果 | `eval/`（`dev/` + `test/`） | 否 |

> 当前阶段：**不训练基础模型**。`train/` 保持为空。Reumani 目前是"现成大模型 + RAG + 工具编排"。

## 评测隔离（最重要）

**最终测试集 `eval/test/` 是冻结的，开发期间绝不允许接触。** 具体地，`test/` 里的样本
**不得**参与以下任何一项：
- 微调 / 训练
- 提示词（system prompt）优化
- 同义词词典设计（`retrieval.SYNONYMS`）
- 阈值 / 参数选择（RRF `k`、`pool`、`top_k`、重排序模型选择等）
- 任何人工调参、看着分数改代码

**允许在 `eval/dev/` 上调参**；改完后**只在 `test/` 上跑一次**报告最终数字。
若因迭代多次看了 `test/`，该测试集即视为"已烧毁"，需换新的 held-out 集。

## 反污染防火墙

1. **知识库 ≠ 评测标签**：`data_lake/` 里不得放入 `test/` 的相关性金标准标注。
2. **评测查询 ≠ 词典来源**：设计 `SYNONYMS` / 阈值时，不得照着 `test/` 的查询来编。
   （历史教训见 `AUDIT.md` F1：早期 `retrieval_eval.py` 的查询几乎全部由同义词键构成，属自测自己。）
3. **语料快照**：报告评测数字时必须记录 `data_lake` 的语料快照 id / 日期；语料在长，
   不同日期的分数不可直接比较（见 `AUDIT.md` F6）。
4. **人写金标准**：相关性判定优先人工标注的 PMID 金标准；程序化"银标准"只能作探索，且须注明。

## 分阶段推进顺序（不得跳步，各自独立 commit/PR）

```
修复可靠性 → 建立金标准评测 → 文献与证据分层 → 结构化自动化
→ 强沙箱与可复现 → 独立科学验证 → （最后才考虑）训练
```

先审计，再按独立 PR 分阶段实施；不要一次做完所有阶段。
